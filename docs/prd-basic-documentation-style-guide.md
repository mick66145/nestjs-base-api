# PRD 文件撰寫風格指南

本指南定義產品需求文件（PRD）的標準結構與撰寫規範，確保團隊產出一致、清晰且完整的需求文件。

---

## 文件結構總覽

PRD 文件應依序包含以下章節：

| 順序 | 章節 | 必要性 | 說明 |
|------|------|--------|------|
| - | 文件元資訊 | 必要 | 版本、日期、負責人、狀態 |
| 1 | 概述（Overview） | 必要 | 背景、目標、範圍 |
| 2 | 使用者角色（User Roles） | 必要 | 系統相關角色定義 |
| 3 | 使用者故事（User Stories） | 必要 | 功能需求的使用者視角描述 |
| 4 | 功能需求（Functional Requirements） | 必要 | 系統行為與邏輯定義 |
| 5 | 介面需求（UI/UX Requirements） | 必要 | 設計稿連結與頁面說明 |
| 6 | 驗收標準（Acceptance Criteria） | 必要 | 功能完成的驗證條件 |
| 7 | 非功能需求（Non-Functional Requirements） | 必要 | 效能、安全性等品質要求 |
| 8 | 相依系統（Dependencies） | 選用 | 外部系統整合說明 |
| 9 | 里程碑（Milestones） | 選用 | 開發階段規劃 |
| 10 | 附錄（Appendix） | 選用 | 術語定義、參考資料 |

---

## 文件元資訊

文件開頭須包含元資訊表格：

```markdown
# [功能名稱] - 產品需求文件（PRD）

| 文件資訊 |            |
| -------- | ---------- |
| 文件版本 | v1.0       |
| 建立日期 | YYYY-MM-DD |
| 負責 PM  | [PM 姓名]  |
| 狀態     | Draft      |

---
```

**狀態值定義：**
- `Draft`：草稿階段，尚未定稿
- `In Review`：審查中
- `Approved`：已核准，可進入開發
- `Deprecated`：已棄用

---

## 1. 概述（Overview）

### 1.1 背景與目的

說明專案的商業背景與建置目的，應回答：
- 為什麼要做這個功能？
- 解決什麼問題？
- 目標使用者是誰？

### 1.2 專案目標

使用表格呈現目標與可衡量指標：

```markdown
| 目標 | 衡量指標 |
| ---- | -------- |
| 目標描述 | 具體且可量化的指標 |
```

**指標撰寫原則：**
- 必須具體可量化
- 避免模糊描述（如「提升體驗」）
- 可包含數字目標（如「< 2 秒」、「減少 80%」）

### 1.3 專案範圍

明確定義功能邊界：

```markdown
**包含（In Scope）：**
- 項目 1
- 項目 2

**不包含（Out of Scope）：**
- 排除項目 1（原因說明）
- 排除項目 2（由 XX 系統處理）
```

## 2. 使用者角色（User Roles）

以表格列出所有相關角色：

```markdown
| 角色 | 說明 |
| ---- | ---- |
| 角色名稱 | 該角色在系統中的職責與權限說明 |
```

**撰寫原則：**
- 使用業務端熟悉的角色名稱
- 說明應包含該角色的主要職責
- 若角色有層級關係，應說明清楚

---

## 3. 使用者故事（User Stories）

### 3.1 User Story 格式

使用標準表格格式：

```markdown
| ID | 角色 | User Story | 優先級 |
| -- | ---- | ---------- | ------ |
| US-001 | 角色名稱 | 作為[角色]，我希望能[功能描述]，以便[目的/價值] | P0 |
```

### 3.2 ID 編號規則

- 格式：`US-[區塊編號][流水號]`
- 區塊編號以百位數區分功能區塊
- 範例：
  - `US-001` ~ `US-099`：基礎設定功能
  - `US-101` ~ `US-199`：核心流程功能
  - `US-1A01`：擴充子區塊（A/B/C...）

### 3.3 優先級定義

| 優先級 | 定義 | 說明 |
| ------ | ---- | ---- |
| P0 | 必須（Must Have） | MVP 必要功能，無此功能系統無法運作 |
| P1 | 應該（Should Have） | 重要功能，首次發布時應包含 |
| P2 | 可以（Could Have） | 加分功能，時間允許時開發 |
| P3 | 未來（Won't Have） | 未來版本規劃，本次不開發 |

### 3.4 按功能區塊分組

將 User Stories 按功能模組分組，每組使用獨立子標題：

```markdown
### 3.1 [功能區塊名稱]

| ID | 角色 | User Story | 優先級 |
...

### 3.2 [另一功能區塊]
...
```

### 3.5 詳細流程與卡控規則（選用）

對於複雜流程，可新增子章節（如 3A、3B）詳述：

#### 流程圖

使用 ASCII 流程圖說明步驟：

```markdown
```
Step 1：步驟名稱
  └─ 子步驟說明
  └─ 子步驟說明

Step 2：步驟名稱
  └─ 子步驟說明
```
```

#### 卡控規則表

```markdown
| 卡控項目 | 規則說明 | 錯誤訊息 |
| -------- | -------- | -------- |
| 檢核名稱 | 具體的檢核邏輯 | 使用者看到的錯誤訊息 |
```

#### 商業規則表

```markdown
| 情境 | 系統行為 |
| ---- | -------- |
| 觸發條件描述 | 系統應執行的動作 |
```

### 4.1 類型定義

```markdown

| 類型 | 說明 | 範例 |
| ---- | ---- | ---- |
| 類型代碼 | 類型說明 | 實際應用範例 |
```

### 4.2 流程邏輯

使用簡化流程圖說明系統行為：

```markdown
```
觸發動作 → 系統檢查 → 條件判斷 → 結果動作
```
```

### 4.3 計算規則

若涉及計算邏輯，以表格說明：

```markdown
| 情境 | 計算規則 |
| ---- | -------- |
| 觸發條件 | 計算公式或邏輯說明 |
```

### 4.4 狀態定義

```markdown
| 狀態 | 說明 |
| ---- | ---- |
| 狀態名稱（Status Code） | 狀態含義與觸發條件 |
```

## 6. 驗收標準（Acceptance Criteria）

### 6.1 AC 格式

```markdown
| AC ID | 驗收條件 |
| ----- | -------- |
| AC-001 | 主詞 + 動作 + 預期結果 |
```

### 6.2 ID 編號規則

- 格式：`AC-[三位數流水號]`
- 按功能區塊分組，編號區間對應 User Story 區塊

### 6.3 撰寫原則

- 使用明確的動詞（可、能、應、顯示、變更）
- 避免模糊詞彙（正確地、適當地）
- 每條 AC 只驗證一件事
- 可被測試驗證

**良好範例：**
- ✓ 顧客取消預約後，票券狀態恢復為「可用」
- ✓ 預約人數達到容量上限時，報名按鈕顯示「已額滿」且無法點擊

**不良範例：**
- ✗ 系統應正確處理取消流程
- ✗ 介面應有良好的使用者體驗

## 7. 非功能需求（Non-Functional Requirements）

```markdown
| 項目 | 需求 |
| ---- | ---- |
| 效能 | 具體指標（如：回應時間 < 2 秒） |
| 可用性 | 具體指標（如：99.5% 以上） |
| 相容性 | 支援的平台/版本 |
| 安全性 | 合規要求或安全標準 |
| 擴展性 | 預期承載量 |
```

## 撰寫風格規範

### 標題格式

- 使用繁體中文標題
- 標題後可加英文對照（如：`概述（Overview）`）
- 章節編號使用阿拉伯數字

### 表格格式

- 所有表格須有表頭
- 表格內容對齊
- 欄位名稱簡潔明確

### 文字風格

- 使用繁體中文
- 專有名詞保持一致性
- 避免冗長句子，每句話聚焦一個概念
- 使用條列式呈現多項內容

### 引用與標註

- 使用 `>` 區塊引用重要說明或備註
- 使用 `**粗體**` 標示重點
- 使用 `「」` 標示系統中的狀態值或 UI 文字
- 使用反引號標示程式碼或 ID

### 版本控管

- 文件內容異動時須更新版本號
- 重大變更時記錄變更歷史
